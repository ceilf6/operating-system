## 文件系统

### 文件树结构

了解文件系统中文件和目录的分布规则是必要的，以便：

- 制定备份方案；
- 管理安全性；
- 在排障时高效操作；
- 安装那些不以软件包形式提供的软件；
- 等等。

根目录是 `/`。

UNIX 系统定义了多种类型的文件：普通物理文件、目录、链接（符号链接或硬链接）、虚拟文件（仅存在于内存中，并包含系统信息）、设备文件。

**FHS**（Filesystem Hierarchy Standard，**文件系统层次结构标准**）为 Unix 系统中的文件系统组织提出了标准化方案。

### 文件树结构（常见目录）

- `/bin`：主要命令
- `/dev`：设备文件
- `/etc`：配置文件与启动脚本
    
    像公司可能会通过 /etc/hosts 重定向限制你的访问
    
- `/sbin`：主要系统命令
- `/home`：用户目录与项目目录
- `/mnt`：临时挂载点（cdrom、floppy 等）
- `/proc`：特殊进程——内核状态
- `/tmp`：临时文件
- `/usr/bin`：非核心用户命令
- `/usr/include`：编译器头文件
- `/usr/lib`：静态库与共享库
- `/usr/local`：本地安装软件，对应于 `/usr`
- `/usr/sbin`：非核心系统命令
- `/usr/share`：可在多个系统之间共享的文件
- `/usr/share/man`：手册页（也可能是 `/usr/man`）
- `/var`：与本机相关的文件（系统与应用）
- `/var/spool`：邮件、打印机、周期性任务等的存放区

### 示例

- **从系统中任意位置访问 `/home` 目录的绝对路径是什么？**
    
    绝对路径是 `/home`
    
- **使用 `pwd` 命令显示当前目录，并记下输出路径。**
    
    命令：`pwd`
    
    示例输出：`/home/username`
    
- **切换到 `/etc` 目录并列出其中的文件。**
    
    `cd /etc`
    
- **不使用绝对路径，返回你的个人主目录。**
    
    命令：`cd ~` 或直接 `cd`
    
- **在你的主目录中创建一个名为 `test_dir` 的目录。**
    
    `mkdir ~/test_dir`
    
- **在该目录中创建三个文件：`file1.txt`、`file2.txt`、`file3.txt`。**
    
    ```bash
    cd ~/test_dir
    touch file1.txt file2.txt file3.txt
    ```
    
- **将 `file2.txt` 重命名为 `renamed_file.txt`。**
    
    **`mv** file2.txt renamed_file.txt`
    
- **删除 `file3.txt` 和目录 `test_dir`。**
    
    ```bash
    rm file3.txt
    cd ..
    rmdir test_dir
    ```
    

### 示例（权限）

- **在你的主目录中创建文件 `example.txt`。**
    
    `touch ~/example.txt`
    
- **使用 `ls -l` 查看它当前的权限。**
    
    `ls -l ~/example.txt`
    
    示例输出：`-rw-r--r--`
    
- **为该文件设置如下权限：**
    - 用户：读、写、执行
    - 组：只读
    - 其他用户：无权限
    
    命令：`chmod 740 ~/example.txt`
    
- **检查新的权限。**
    
    `ls -l ~/example.txt`
    
    示例输出：`-rwxr-----`
    
- **尝试以另一个用户身份访问该文件，并记录结果。**
    
    结果：`Permission denied`（权限被拒绝）

## 用户账户

### Unix 用户

一个 Unix 用户由以下属性描述：

- 登录名（login）
- 密码
- 唯一的数字标识符（UID）
    
    user ID
    
- 主组（**G**ID）
    
    group
    
- 注释信息（称为 gecos）
- 用户主目录（home directory）
- 默认命令解释器（shell）

系统通过 UID 识别用户。**`root` 用户的 UID 为 0**。正是这一特性赋予它对系统的完全访问权限。

这些信息都以文本格式存储在 **`/etc/passwd`** 文件中，各字段之间用 `:` 分隔。

例如：

```bash
cat /etc/passwd
```

```bash
root:*:0:0:System Administrator:/var/root:/bin/sh
```

```
utseus:x:500:500:cours lo03:/home/utseus:/bin/bash
```

### `/etc/passwd` 的格式

### 一些建议

- 应始终按照 **UID 的数值顺序**对 `/etc/passwd` 的行进行排序：**`sort** -t : +2n -3 /etc/passwd`
- 绝不能保留没有密码的账户。

### 第 1 列：login

计算机识别个人所使用的名称。通常为 8 个字符；若更多需确认。

在 Linux 中，用户名可以超过 8 个字符。

建议：不要使用大写字母，避免使用带重音的字符。

### 第 2 列：密码

用户密码以加密且**不可逆的形式存储**。

> 应该是 hash 化？这也就是为什么只能重设密码
> 

有必要教育用户选择高质量密码。

密码破解软件并不是“解密”密码，而是通过字典尝试进行**猜测**。

### 第 3 列：UID

取值通常在 0 到 32767 之间（理论上是 short int）。

在 Linux 中，它是 `uid_t`，即无符号整数。

该值在所有用户中必须**唯一**。

### 第 4 列：GID

取值通常在 0 到 32767 之间（理论上是 short int）。

在 Linux 中，它是 `gid_t`，即无符号整数。

该值在所有组中必须**唯一**。

示例：

```
% ls -ln
-rw-r--r-- 1 4332 1000 5544 Sep 15 22:47 cours.dvi
-rw-r--r-- 1 4332 1000 940  Sep 15 22:27 cours.tex

% ls -l
-rw-r--r-- 1 utseus software 5928 Sep 15 22:49 cours.dvi
-rw-r--r-- 1 utseus software 940  Sep 15 22:27 cours.tex
```

### 第 5 列：gecos

用户的**明文**身份信息。

BSD 系统也会在这里存储其他信息，如电话号码、办公室号等。

**`chfn`**（change finger）命令可以修改这一字段。

### 第 6 列：homedir

默认目录，即**用户主目录**。

### 第 7 列：shell

默认 shell。

如果该字段为空，则默认使用 **`/bin/sh`**。

shell 会决定应在用户目录中安装哪些配置文件。

相关命令：**`chsh`**

### 创建新账户

创建一个账户时通常需要完成以下步骤：

1. 根据用户所属部门选择 UID 和 GID。
2. 选择主目录（可能与部门有关）。
3. 按照本地策略选择登录名。
4. 选择登录 shell。
5. 将这些信息加入**账户数据库**（**`/etc/passwd`** 或 **NIS**，取决于是否集中管理和本地策略）；默认密码字段通常先设为 `x`。
6. 在 `/etc/**group**` 中添加该用户。
7. 创建主目录。
8. **复制环境配置文件**（**`.profile`、`.cshrc`、`.xsession`** 等）。
9. 使用 `chown` 和 `chgrp` 将新建主目录归属给该用户。
10. 如果用户在场，初始化密码。

在自动化创建账户时，最棘手的一步当然是向账户数据库中插入一条新记录。

### root 账户

它的特殊性来自其 **UID = 0**。

一些规则：

1. `root` 用户的 **`PATH` 中不应包含 `.`**（避免**本地命令优先于系统命令**）。
    
    例如假如本地有一个 rm 如果 PATH 中有 . 会导致本地 rm 优先于系统的 rm
    
2. `root` 用户的 `umask` 应设为 `022`（某些文件必须能被普通用户**访问**）。
    
    > umask 是 屏蔽权限，控制组用户和其他用户不可写
    > 
3. 应避免把 `/` 作为 `root` 的主目录（否则配置文件会污染根目录）。
    
    root 的 / 下面就是根目录，里面放着一系列关键目录，例如 /etc /bin 等等
    

### 虚拟账户

虚拟账户本质上主要是**文件的所有者**。

示例：

```
root:x:0:0:root:/root:/bin/bash
bin:x:1:1:bin:/bin:
daemon:x:2:2:daemon:/sbin:
adm:x:3:4:adm:/var/adm:
lp:x:4:7:lp:/var/spool/lpd:
sync:x:5:0:sync:/sbin:/bin/sync
shutdown:x:6:0:shutdown:/sbin:/sbin/shutdown
halt:x:7:0:halt:/sbin:/sbin/halt
mail:x:8:12:mail:/var/spool/mail:
news:x:9:13:news:/var/spool/news:
```

例如，**`bin` 是大多数可执行文件的所有者**：

```
-rwxr-xr-x 1 bin  bin  40960 Aug  9 1994 stty*
-rws--x--x 1 root bin  24576 Aug  9 1994 su*
-rwxr-xr-x 1 bin  bin  24576 Aug  9 1994 sum*
```

### Unix 组

`/etc/group` 是一个 ASCII 文件，用于定义每个用户所属的组。

每个组占一行，格式如下：

```
组名:密码:GID（唯一数字标识）:用户列表
```

**用户列表可以为空，也可以包含一个或多个用户名，用户名之间以 `,` 分隔**。

登录时，用户会被关联到他所属的所有组中。可以使用 **`id`** 或 **`groups`** 命令查看：

```
# id
uid=501(utseus) gid=501(utseus) groupes=501(utseus),100(users),101(cvs),200(compta)

# groups
utseus users cvs compta
```

### UID 的约定

用户列表一般按如下方式划分：

- UID 为 **0** 的用户是 `root`
- **UID 小于某个阈值的用户是系统/虚拟用户**
- UID 在该阈值与 65533 之间的是普通真实用户
- UID 为 65534 的用户通常是 `nobody` 或 `nfsnobody`（在支持 32 位 UID 的系统中，用于兼容旧 NFS 服务器）

通常，每个用户会对应一个组，并且：

- **该组的 GID 与用户的 UID 相同**；
- 组名与用户名相同；
- 该组是该用户的主组。

支持 32 位 UID 的用户机制已开始在较新的发行版中出现，使可支持的用户数量从 65535 提升到 400 多万。

### 隐藏密码（shadow）

起初，密码存储在 `/etc/passwd` 文件中。

密码是以加密形式存储的，在数学上不可逆。

由于 `/etc/**passwd**` 必须对所有用户**可读**（因为其中包含 UID、主目录、shell 等信息），密码存储后来迁移到了读取权限**受限**的 **`/etc/shadow`** 文件中。

**MD5** 编码算法越来越常用。

> 果然就是哈希化
> 

此外，该文件还加入了额外字段（以“**天**”为单位），用于管理密码**过期时间**。

例如：

```
utseus:c8NuTn3ScFJfA:12460:0:99999:7:::
```

其中：

- 第 4 个字段 `0`：最短有效期
- 第 5 个字段 `99999`：最长有效期

而 `/etc/passwd` 中原本用于存放密码的字段通常会被设置为 `x`。

### 用户与组管理

创建一个用户通常包括以下几个步骤：

- 在 `passwd`、`shadow` 和 `group` 文件中创建用户，以及在需要时创建其主组；
- 创建用户主目录（通常位于 `/home`），并赋予正确权限；
- 如有需要，为用户分配磁盘配额；
- 从 `/etc/skel` 复制一些标准文件到用户主目录，并设置合适的权限。

这些操作可以通过以下方式完成：

- **手动方式**：使用 `vi`、`mkdir`、`cp`、`chown`、`chfn`、`chsh` 等，不太推荐；
- **命令行方式**：主要使用 `adduser`；
- **图形化方式**：如 Webmin（几乎可以通过 Webmin 管理一切）或 RedHat 的用户管理工具。

用户修改、删除以及用户组管理的原理也是一样的。

### 示例（用户与组管理）

- **在 `/etc/passwd` 中识别以下账户：root、一个系统账户（如 daemon）以及一个普通用户账户（`username:x:UID:GID:comment:home_directory:shell`）**
    - root 账户：UID = 0，GID = 0
    - 系统账户 daemon：UID = 1，GID = 1
    - 普通用户账户：UID ≥ 1000
    例如：`user:x:1001:1001::/home/user:/bin/bash`
- **密码列**`/etc/passwd`**中的字符 `x` 表示什么？**
    
    **表示密码出于安全原因存储在 `/etc/shadow` 中**。
    
- **创建一个名为 `testuser` 的用户并为其创建主目录。**
    
    `sudo **useradd** -m testuser`
    
- **给 `testuser` 添加一段描述：“测试用户”。**
    
    `sudo **usermod** -c "测试用户" testuser`
    
- **将 `testuser` 的默认 shell 改为 `/bin/zsh`。**
    
    `sudo **usermod** -s /bin/zsh testuser`
    
- **删除 `testuser` 及其主目录。**
    
    `sudo **userdel** -r testuser`
    
- **为账户 `guest` 设置一个过期日期。**
    
    `sudo chage -E 2024-12-31 guest`
    
- **将 `testuser` 的密码改为 `securepass123`。**
    
    `sudo passwd testuser`，然后输入 `securepass123`
    
- **锁定 `testuser` 账户，禁止登录。**
    
    `sudo passwd -l testuser`
    
- **解锁 `testuser` 账户。**
    
    `sudo passwd -u testuser`
    
- **如果用户在账户被锁定时尝试登录，会发生什么？**
    
    结果：`Account is locked.`（账户已锁定）
    
- **如果两个用户拥有相同的 UID，会发生什么？**
    
    它们会共享相同**权限**，并在系统看来像同一个用户，这会引发冲突。
    
- **列出 `/etc/passwd` 中所有系统账户。**
    
    `cat /etc/passwd | **grep** -E '^[^:]+:x:[0-9]{1,2}:'`
    
- **解释 `nobody` 和 `daemon` 账户的作用。**
    - `nobody`：用于那些**不需要特殊权限的服务**
    - `daemon`：用于运行某些**系统服务**
- **为什么系统账户默认没有有效 shell？**
    
    系统账户通常没有有效 shell（如 `/usr/sbin/nologin`），以**避免交互式登录并降低安全风险**。

## 任务调度

### cron 任务计划器

任务计划器 `cron` 允许每个用户通过 **`crontab`** 命令，配置程序或脚本在**指定时间执行**，或者周期性执行。

`root` 用户也可以这样做。不过，如果是安装与系统管理相关的周期性任务，更推荐使用以下目录：

- `/etc/cron.hourly`：每小时执行
- `/etc/cron.daily`：每天执行
- `/etc/cron.weekly`：每周执行
- `/etc/cron.monthly`：每月执行
- `/etc/cron.d`：用于更精细的计划配置

前四个目录的使用很简单：只需把一个可执行文件复制进去（或创建一个链接），即可启用其周期性执行。

### `/etc/cron.d` 的使用

`/etc/cron.d` 目录中的文件更特殊一些。该目录中的文件是文本文件，其格式与 `crontab` 所操作的文件几乎相同。

例如：

```
MAILTO=root
#mn hr day month dow user command
0 4 * * * postgres /usr/lib/postgresql/bin/do.maintenance -a
```

表示：每天凌晨 4:00，以 `postgres` 用户身份执行该命令。

如果只想在工作日执行，则星期字段可写为 `1-5`。

`/etc/crontab` 文件控制 `/etc/cron.*` 目录中命令的执行日期和时间（但不包括 `/etc/cron.d`）。

用户自己创建的计划任务通常以“每个用户一个文件”的形式存储在 `/var/spool/cron` 目录中。

### 示例（cron）

- **下面这条 crontab 记录的作用是什么？**
    
    ```
    **30 2** * * 1-5 /usr/local/bin/backup.sh **>**> /var/log/backup.log 2>&1
    ```
    
    ```bash
    分钟 小时 天 月 周 执行文件
    ```
    
    含义：
    
    - 在 **2:30**
    - **每月的每天**
    - **每个月**
    - **周一到周五**
    执行 `/usr/local/bin/backup.sh`，并把标准输出和错误输出都追加到（管道化） `/var/log/backup.log`。
- **安排一个任务：让 `/usr/bin/monitor.sh` 每天 7:15 执行一次，并在每周一和周四 14:45 再执行一次。**
    
    ```
    15 7 * * * /usr/bin/monitor.sh
    45 14 * * 1,4 /usr/bin/monitor.sh
    ```
    
- **如果 `30 8 * * * /home/user/script.sh` 没有执行，应该如何排查？**
    - 检查 cron 是否在运行：`systemctl status cron`
    - 确认脚本可执行：**`chmod** +x /home/user/script.sh`
    - 手动测试脚本：`/home/user/script.sh`
    - 查看 cron 日志（某些系统中）：`grep CRON /var/log/syslog`
    - 脚本可能依赖**环境变量**，可**在 cron 任务前添加**：
        
        ```
        SHELL=/bin/bash
        PATH=/usr/local/sbin:/usr/local/bin:/sbin:/bin:/usr/sbin:/usr/bin
        30 8 * * * /home/user/script.sh
        ```
        
- **检查语法以避免错误。**
    
    `crontab -l`
    
- **cron 和 at 的区别是什么？**
    - `cron`：按预定日历**反复执行**任务
    - `at`：在某一个指定时间**执行一次**任务
- **给出一个 `at` 命令的使用示例。**
    
    ```bash
    echo "/home/user/script.sh" | at 10:00 tomorrow
    ```
