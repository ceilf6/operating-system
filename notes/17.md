`1, 5, 2, 5, 1, 4, 1, 5, 3`

### 算法 A1

| 帧1 | 1 | 1 | 1 | 1 | 1 | 4 | 4 | 4 | 3 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| 帧2 |  | 5 | 5 | 5 | 5 | 5 | 1 | 1 | 1 |

## 8. 死锁（L’interblocage）

设：

- `S1, S2, S3, S4` 为信号量；
- `S1 = S2 = S3 = 1`
- `S4 = 0`

5 个进程定义如下：

### 进程 A

```
开始
  P(S1) ;
  V(S1) ;
  P(S2) ;
  V(S2)
结束
```

### 进程 B

```
开始
  P(S1) ;
  P(S2) ;
  V(S4) ;
  V(S2) ;
  V(S1)
结束
```

### 进程 C

```
开始
  P(S2) ;
  P(S3) ;
  V(S2) ;
  V(S3)
结束
```

### 进程 D

```
开始
  P(S4) ;
  P(S2) ;
  P(S1) ;
  V(S1) ;
  V(S2)
结束
```

### 进程 E

```
开始
  P(S3) ;
  P(S2) ;
  V(S2) ;
  V(S3)
结束
```

这 5 个进程 `A、B、C、D、E` 可以按任意顺序在 **5 个不同处理器**上启动。

信号量 `S1、S2、S3、S4` 用于管理对 **4 类不同资源**的访问。

1. 在上述 5 个进程中，哪些进程会导致**死锁**？
2. 由于前述阻塞，哪些**额外进程**也可能被阻塞？
3. 对于给定的这些进程，死锁是**必然发生**，还是取决于任务执行的先后次序？

## 个人答案：

逻辑分析

A 占用和释放是相邻匹配的，相对安全

B 占用 S1 等 S2

C 占用 S2 等 S3

D 注意 S4 初始值为 0 所以一开始必阻塞，只有等 B 释放 S4 才能继续

E 占用 S3 等 S2

### 1

看到上面 C 和 E 的占用和等待是互补的，所以这两个可能会导致死锁

### 2

B 等待 S2 但是被 C 占用了

D 等待 B 的 S4

A 也等待 S2

### 3

依赖于任务执行顺序

C 先执行完然后 E → B … 关键是不要让有死锁风险的进程并行

## 9. 安全状态还是不安全状态？（État sûr ou pas ?）

考虑一个系统，包含 4 个进程 `A、B、C、D`，以及 3 类资源 `R、S、T`。

系统中资源总量如下：

- `R` 有 **3 个实例**
- `S` 有 **2 个实例**
- `T` 有 **2 个实例**

当前资源分配与请求情况如下：

- **A** 持有 1 个 `R`，并请求 1 个 `S`
- **B** 持有 2 个 `S`，并请求 1 个 `R` 和 1 个 `T`
- **C** 持有 1 个 `R`，并请求 1 个 `S`
- **D** 持有 2 个 `T`，并请求 1 个 `R`

要求：

1. 构造该系统的**资源分配图（graphe d’allocation des ressources）**。
    
    资源指向进程实线占用、进程指向资源虚线申请
    
    > 无端想到了 “卡坦岛“
    > 
    
    就按照上面的信息画即可
    
    1. 分配边
    
    从资源指向进程：
    
    - R -> A
    - R -> C
    - S -> B（两条，因为 B 持有 2 个 S）
    - T -> D（两条，因为 D 持有 2 个 T）
    1. 请求边
    
    从进程指向资源：
    
    - A -> S
    - B -> R
    - B -> T
    - C -> S
    - D -> R
2. 是否存在**死锁**？如果有，涉及哪些进程？
    
    用 **图归纳法检测死锁**，开始消消乐
    
    一般 后面进行的进程 是 前面完成的进程释放资源的请求者
    
    D → B → A → C
    
3. 使用**银行家算法（algorithme du banquier）验证该状态是否为安全状态**。
    
    假设性原则
    
    ```bash
    R=3, S=2, T=2
    
    Alloc : 已分配
    A = (1,0,0)
    B = (0,2,0)
    C = (1,0,0)
    D = (0,0,2)
    
    Req : 需求
    A = (0,1,0)
    B = (1,0,1)
    C = (0,1,0)
    D = (1,0,0)
    
    当前可用 = total - Alloc
    Available = (1,0,0)
    
    Banquier :
    - D peut terminer, Work=(1,0,2)
    - B peut terminer, Work=(1,2,2)
    - A peut terminer, Work=(2,2,2)
    - C peut terminer, Work=(3,2,2)
    
    Séquence sûre : D -> B -> A -> C
    
    Donc il n’y a pas d’interblocage et l’état est sûr.
    ```
