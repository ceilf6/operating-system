解释 `syslog.conf` 中的以下几行：

4. **`mail,uucp.notice;uucp.!=alert /var/log/mail`**
    
    来自 **mail** 和 **uucp** facilities、优先级为 **notice 及更高** 的消息会被写入 `mail` 文件，
    
    但 **uucp 中优先级恰好为 alert 的消息除外**。
    
5. **`mail,uucp.notice;uucp.!alert /var/log/mail`**
    
    来自 **mail** 和 **uucp** facilities、优先级为 **notice 及更高** 的消息会被写入 `mail` 文件，
    
    但 **uucp 中优先级为 alert 及更高的消息除外**。

## Syslog 与 Rsyslog

`syslog` 和 `rsyslog` 都是 Unix/Linux 下的日志管理与处理系统。

它们具有一些基础的共同功能，但在能力和使用场景上有所不同。

### Rsyslog

`rsyslog` 是 `syslog` 的增强版本，用来弥补后者的限制，并满足现代环境的需求。

- **向后兼容**：
    
    可以使用 `syslog` 的配置（`/etc/syslog.conf`），但通常使用 `/etc/rsyslog.conf`。
    
- **更先进的协议支持**：
    - 支持现代 Syslog 协议，如 **RFC 5424**（比 RFC 3164 更健壮）。
    - 支持 **UDP、TCP 和 TLS**，可实现安全通信。
- **更强大且可扩展**：
    - 能处理海量日志。
    - 可将日志写入数据库（如 MySQL、PostgreSQL），或发送到集中式日志系统。
- **高级过滤能力**：
    - 支持基于正则表达式或特定条件的复杂规则。
- **额外模块**：
    - 可通过模块增加功能（例如 `ommysql` 用于 MySQL）。
- **性能更高**：
    - 针对高强度日志环境做了优化。
- **结构化日志**：
    - 支持 JSON 等格式，适合现代集成场景。
- **集中式管理**：
    - 在分布式环境下常用于集中采集日志。

### 什么时候用 syslog，什么时候用 rsyslog？

- **Syslog**：
    
    如果你的系统比较简单、轻量，日志需求也比较基础。
    
- **Rsyslog**：
    
    如果你的环境更复杂、日志量更大，或需要与数据库、现代系统集成。
    

> 在大多数现代场景中，更推荐使用 **rsyslog**，因为它更强大、更灵活，同时仍保持与 syslog 的兼容性。
>

# 练习 4 - 配置 rsyslog 将日志发送到远程服务器

## 题目

配置一台 Unix 服务器，将**所有日志消息**发送到一个远程服务器，目标地址为 **192.168.1.100**，使用 **UDP 514** 端口。

## 答案

在客户端打开 `/etc/rsyslog.conf`，加入下面这行，将日志发送到远程服务器：

```
*.* @192.168.1.100:514
```

如果你想使用 **TCP**（比 UDP 更可靠，因为 UDP 不保证消息一定送达），把 `@` 改成 `@@`：

```
*.* @@192.168.1.100:514
```

---

本练习关注的是**集中式日志管理**：通过配置 Unix/Linux 系统（客户端），把它产生的所有日志消息转发到专门的远程日志服务器。集中式日志对于**安全分析、审计、以及统一排查多台机器问题**都非常重要。

![image.png](attachment:cfd9d263-f31c-4cdd-b1fc-e34ac886f3c5:image.png)

## 详细解释

### 1. 目标

核心目标是**重定向本机的日志流**。也就是说，系统不仅可以像平时一样把日志写到本地文件（例如 `/var/log/syslog`），还会让 `rsyslog` 守护进程把每一条日志通过网络发送到 IP 为 `192.168.1.100` 的远程服务器。

### 2. 配置文件

`rsyslog` 的主要配置文件通常是：

```bash
/etc/rsyslog.conf
```

这个文件里定义了各种规则，用来说明：

- 哪些日志消息会被匹配
- 按照什么条件匹配（facility 与 severity）
- 匹配后要发送到哪里（文件、控制台、远程主机等）

### 3. 转发规则语法

这一行：

```
*.* @192.168.1.100:514
```

就是一条**日志转发规则**。

- `.*`
    - 第一个  表示 **facility（日志来源子系统）**
        - 如 `auth`、`kern`、`mail`
    - 第二个  表示 **severity / priority（严重级别）**
        - 如 `emerg`、`alert`、`crit`、`err`、`warning`、`notice`、`info`、`debug`
    - 所以 `.*` 的意思就是：**所有 facility 的所有级别日志**
- `@`
    - 单个 `@` 表示使用 **UDP** 协议发送
- `192.168.1.100:514`
    - 目标服务器地址与端口
    - `192.168.1.100` 是远程日志服务器 IP
    - `514` 是 syslog 协议默认端口

### 4. UDP 与 TCP 的区别

这是一个很关键的点。

### UDP（`@`）

- UDP 不保证消息送达
- 它是无连接协议，属于“发出去就不管了”的方式
- 如果网络拥塞，日志可能丢失，而发送端并不会知道
- 优点是开销小、速度快
- 但缺点是**不可靠**
    
    > 像 HTTP3.0 从之前用 TCP 转为 UDP 就是因为其快
    > 

配置示例：

```
*.* @192.168.1.100:514
```

### TCP（`@@`）

- TCP 会先建立连接
- 它会通过确认机制和重传机制保证数据送达
- 在日志收集场景中，如果像“认证失败”或“磁盘已满”这种关键日志丢失，后果可能很严重
- 所以在生产环境下，TCP 通常更可靠

配置示例：

```
*.* @@192.168.1.100:514
```

### 5. 操作步骤

### 编辑客户端配置文件

```bash
sudo nano /etc/rsyslog.conf
```

### 在规则部分末尾添加一行

- 用 UDP：

```
*.* @192.168.1.100:514
```

- 用 TCP：

```
*.* @@192.168.1.100:514
```

### 重启 rsyslog 服务使配置生效

```bash
sudo systemctl restart rsyslog
```

# 练习 5 - 过滤 warning 级别消息并传给 Bash 脚本处理

## 题目

配置 `rsyslog`，检测所有来自 **auth facility** 且级别为 **warning（或更高）** 的日志消息，并把它们发送给一个 Bash 脚本。

这个脚本需要把日志写入文件：

```bash
/var/log/auth_warning.log
```

并且格式如下：

```
2024-12-30 14:35:42 Authentication failure for user 'admin'.
```

![image.png](attachment:ed7dab5b-ef15-4446-9f02-63935ea87dc4:image.png)

## 答案

创建脚本文件 `/usr/local/bin/auth_warnings.sh`：

```bash
#!/bin/bash
while read -r line; do
  echo "$(date '+%Y-%m-%d %H:%M:%S') $line" >> /var/log/auth_warning.log
done
```

给它加执行权限：

```bash
sudo chmod +x /usr/local/bin/auth_warnings.sh
```

然后修改 `/etc/rsyslog.conf`：

### 写法 1：传统管道语法

```
auth.warning | /usr/local/bin/auth_warnings.sh
```

### 写法 2：基于属性的旧式过滤

```
if ($syslogfacility-text == 'auth' and $msg contains 'warning') then | /usr/local/bin/auth_warnings.sh
```

### 写法 3：现代 RainerScript + `omprog`（推荐）

```
if ($syslogfacility-text == 'auth' and $syslogseverity <= 4) then {
    action(type="omprog" binary="/usr/local/bin/auth_warnings.sh")
    stop
}
```

---

本练习展示了更高级的日志处理方式：不是简单地把日志写入文件，而是**先按条件过滤，再交给自定义脚本进行进一步处理**。

## 详细解释

### 1. 目标

拦截所有来自 `auth` / 授权相关 facility 的日志消息，且这些消息的严重级别必须是 **warning 或更高**（也就是更严重：`err`、`crit`、`alert`、`emerg`）。

被筛选出来的日志会被持续送入一个 Bash 脚本，由脚本补充时间戳并写入目标文件。

### 2. Facility 与 Severity 的理解

`rsyslog` 中每条日志一般有两个关键属性：

### Facility

表示日志来自哪个子系统，例如：

- `kern`：内核
- `mail`：邮件系统
- `auth`：认证 / 安全

### Severity

表示日志严重程度，编号从 `0` 到 `7`：

- `0 = emerg`
- `1 = alert`
- `2 = crit`
- `3 = err`
- `4 = warning`
- `5 = notice`
- `6 = info`
- `7 = debug`

注意：**数字越小，严重程度越高**。

### 3. Bash 脚本说明

脚本 `/usr/local/bin/auth_warnings.sh` 的作用是一个**简单的处理器**。

它会一直循环，从标准输入中不断读取由 `rsyslog` 传过来的日志行：

```bash
while read -r line; do
  ...
done
```

对每一行日志：

- 在前面加上当前时间戳
- 追加写入 `/var/log/auth_warning.log`

例如：

```bash
echo "$(date '+%Y-%m-%d %H:%M:%S') $line" >> /var/log/auth_warning.log
```

执行：

```bash
chmod +x /usr/local/bin/auth_warnings.sh
```

是为了让脚本具备可执行权限。

### 4. 三种 rsyslog 配置方法

### 方法 1：传统管道语法 `|`

```
auth.warning | /usr/local/bin/auth_warnings.sh
```

含义：

- 选择 `auth` facility
- 严重级别为 `warning` 及以上
- 直接通过管道把日志传给脚本

这种方式简单直观，但功能较弱。

### 方法 2：基于属性的旧式 if 过滤

```
if ($syslogfacility-text == 'auth' and $msg contains 'warning') then | /usr/local/bin/auth_warnings.sh
```

含义：

- 过滤 facility 为 `auth` 的消息
- 并且日志正文里包含字符串 `warning`

这种方式的问题是：它是看**文本内容**，不是看日志真正的**severity 属性**，因此不够精确。

### 方法 3：现代 RainerScript + `omprog`（推荐）

```
if ($syslogfacility-text == 'auth' and $syslogseverity <= 4) then {
    action(type="omprog" binary="/usr/local/bin/auth_warnings.sh")
    stop
}
```

这是更现代、更可靠的方式。

- `if (...)`
    - 判断 facility 是 `auth`
    - 且 severity 数值小于等于 4（即 `warning` 及以上）
- `action(type="omprog" ...)`
    - 使用 `omprog` 模块，把日志发送给外部程序
- `stop`
    - 这一条非常重要
    - 一旦消息命中这条规则，后续规则就不再继续处理
    - 这样可以避免同一条日志又被写到默认系统日志文件里，造成重复记录

### 5. 为什么推荐 `omprog`

`omprog` 是 rsyslog 向外部脚本/程序发送消息的现代输出模块，可以理解成比普通管道更强大的替代品。

### 优点 1：更可靠

脚本处理完每条日志后，可以向 rsyslog 返回确认，降低日志丢失风险。

### 优点 2：更高效

它通常让脚本作为**一个长时间运行的进程**存在，持续接收日志流。

而旧式简单管道往往可能频繁启动脚本实例，效率更低。

### 优点 3：更可控

可以更精细地控制：

- 超时
- 自动重启
- 消息格式
- 异常处理

### 6. 关于 `omprog` 的补充

1. `omprog` 是 Linux 下 `rsyslog` 的一个输出模块
2. 名字含义是 “Output Module Program”
3. 它把日志消息转发给外部应用程序
4. 外部程序通过标准输入 `stdin` 接收日志
5. 它适合做实时日志处理与分析
6. 必须在 rsyslog 规则中配置
7. 配置形式一般是 `action(type="omprog" ...)`
8. 必须提供脚本或程序的完整路径
9. 例如：`action(type="omprog" binary="/path/to/script")`
10. 目标程序通常会持续运行
11. 每一条新日志都会被送到这个等待中的进程
12. 因此可以对特定日志事件立即作出响应
13. 它比“只写文件”更灵活
14. 很适合实现自定义告警和自动化通知
15. 例如收到 “failure” 日志后自动发邮件
16. 程序本身要能处理持续不断的数据流
17. 脚本最好实现正确的信号处理，便于优雅退出
18. 可以借此把日志系统接入其他 IT 系统
19. 让 rsyslog 从被动记录器升级为主动处理工具
20. 所以 `omprog` 是自动化和监控中很有价值的能力

# 练习 6 - 为指定用户监控日志

## 题目

编写一个脚本，用来监控某个指定用户相关的日志：

1. 检查这个用户是否存在于系统中
2. 在日志文件（`/var/log/syslog`）中查找包含该用户名的事件
3. 将结果保存到该用户主目录下的一个文本文件中
4. 使用 `cron` 让这个脚本每天自动执行

## 答案

### 监控脚本

```bash
#!/bin/bash

# 询问要监控的用户名
echo "Entrez le nom de l'utilisateur à surveiller :"
read -r USERNAME

# 检查用户是否存在
if ! id "$USERNAME" &>/dev/null; then
    echo "L'utilisateur $USERNAME n'existe pas."
    exit 1
fi

# 获取用户主目录
USER_HOME=$(getent passwd "$USERNAME" | cut -d: -f6)

# 检查主目录是否存在
if [ ! -d "$USER_HOME" ]; then
    echo "Le répertoire de l'utilisateur $USERNAME est introuvable."
    exit 1
fi

# 保存结果的文件
EVENTS_FILE="$USER_HOME/user_events.log"

# 在 /var/log/syslog 中搜索包含用户名的事件
echo "Événements pour l'utilisateur $USERNAME (exécuté le $(date)) :" > "$EVENTS_FILE"
grep "$USERNAME" /var/log/syslog >> "$EVENTS_FILE"

echo "Les événements ont été enregistrés dans $EVENTS_FILE"
```

### 定时执行（cron）

编辑当前用户的 crontab：

```bash
crontab -e
```

加入一行：

```bash
0 0 * * * /home/utilisateur/scripts/monitor_user_simple.sh > /home/utilisateur/scripts/monitor_user_simple.log 2>&1
```

![image.png](attachment:377694cf-076e-45a4-ac40-bc43123626cd:image.png)

本练习结合了 **Shell 脚本** 和 **cron 定时任务**，用于构建一个面向指定用户的自动化日志监控工具。

## 脚本详细解释

### 1. 目标

编写一个脚本：运行时接收一个用户名，去系统日志中查找与该用户名相关的记录，并把结果写到这个用户主目录中的报告文件里。随后，再把该脚本设置成**每天自动执行**。

### 2. 脚本流程解析

### 用户输入

```bash
read -r USERNAME
```

提示操作者输入要监控的用户名。

### 用户是否存在

```bash
id "$USERNAME" &>/dev/null
```

用于检测系统中是否存在该用户：

- 如果 `id` 命令执行失败，说明用户不存在
- 脚本会打印错误并退出

### 获取用户主目录

```bash
getent passwd "$USERNAME" | cut -d: -f6
```

这一行更稳妥地从系统用户数据库中获取主目录。

- `getent passwd "$USERNAME"` 会读取该用户的 passwd 记录
- 用户记录通常形如：

```
login:passwd:uid:gid:gecos:home:shell
```

- `cut -d: -f6` 会取出第 6 列，也就是主目录路径

`getent` 的好处是它不仅支持本地 `/etc/passwd`，也支持 LDAP 等其他用户源。

### 主目录是否存在

脚本还会进一步判断这个主目录路径是否真的存在。

### 定义输出文件

```bash
EVENTS_FILE="$USER_HOME/user_events.log"
```

### 搜索日志

核心命令是：

```bash
grep "$USERNAME" /var/log/syslog >> "$EVENTS_FILE"
```

它会在 `/var/log/syslog` 中搜索包含用户名的每一行，并追加写入结果文件。

在此之前，脚本会先写入一行带有执行时间的标题：

```bash
echo "Événements pour l'utilisateur $USERNAME (exécuté le $(date)) :" > "$EVENTS_FILE"
```

### 注意

这个版本只搜索：

```bash
/var/log/syslog
```

更健壮的版本通常还会同时检查：

- `/var/log/auth.log`
- `/var/log/secure`
- `/var/log/messages`
- 以及 `/var/log/` 下其他相关日志

## `cron` 定时执行详解

### 1. 为什么需要 `cron`

很多系统任务都需要自动、周期性运行，例如：

- 每晚备份数据库
- 每小时清理临时文件
- 定期跑机器学习训练任务
- 持续监控系统健康状态

Linux 内置了一个基于时间的任务调度器：`cron`。

而我们配置它时使用的接口叫做：

```bash
crontab
```

### 2. 概念模型

可以把 `cron` 理解成后台持续运行的一个循环：

```python
while true:
    check current time
    if matches any scheduled job:
        execute command
```

每个用户都可以有自己独立的 `crontab` 文件，用来定义：

- 什么时候执行
- 执行什么命令

### 3. 基本命令

### 查看当前 crontab

```bash
crontab -l
```

### 编辑当前 crontab

```bash
crontab -e
```

### 删除当前 crontab

```bash
crontab -r
```

### 编辑其他用户的 crontab（需要权限）

```bash
sudo crontab -u username -e
```

### 4. crontab 语法

每一行格式如下：

```
* * * * * command
│ │ │ │ │
│ │ │ │ └── 星期几 (0–7)（周日可用 0 或 7）
│ │ │ └──── 月份 (1–12)
│ │ └────── 每月第几天 (1–31)
│ └──────── 小时 (0–23)
└────────── 分钟 (0–59)
```

### 5. 示例

### 每天凌晨 2:30 执行脚本

```bash
30 2 * * * /home/user/backup.sh
```

### 每 5 分钟执行一次

```bash
*/5 * * * * /home/user/script.sh
```

### 每周一上午 9 点执行

```bash
0 9 * * 1 /home/user/report.sh
```

### 开机时执行

```bash
@reboot /home/user/startup.sh
```

### 6. 特殊符号

### 星号

表示“每一个”

```bash
* * * * *    # 每分钟执行
```

### 步长 `/n`

表示“每 n 个单位”

```bash
*/10 * * * *   # 每 10 分钟执行
```

### 范围 `a-b`

```bash
1-5 * * * *    # 每小时的第 1 到第 5 分钟执行
```

### 列表 `a,b,c`

```bash
0 9 * * 1,3,5   # 每周一、三、五 9 点执行
```

### 7. 预定义宏

有时可以不用写完整 5 段时间格式：

- `@yearly` → `0 0 1 1 *`
- `@monthly` → `0 0 1 * *`
- `@weekly` → `0 0 * * 0`
- `@daily` → `0 0 * * *`
- `@hourly` → `0 * * * *`

示例：

```bash
@daily /home/user/daily_job.sh
```

### 8. 环境与执行上下文

经常有人遇到：

> 脚本在终端里能跑，在 cron 里却失败。
> 

原因通常是 `cron` 的运行环境和交互式终端不一样。

### 差异 1：环境变量很少

- 不会自动加载 `.bashrc` 或 `.profile`
- `PATH` 可能很短

解决方式：

```bash
PATH=/usr/local/bin:/usr/bin:/bin
```

### 差异 2：没有交互

- 不会有输入提示
- 不适合依赖用户手动输入

### 差异 3：工作目录不同

默认工作目录通常是用户主目录。

解决方式：

```bash
cd /project && python train.py
```

### 9. 日志与调试

默认情况下，`cron` 不会把输出展示给你看，所以最好显式重定向输出。

示例：

```bash
* * * * * /script.sh >> /tmp/log.txt 2>&1
```

解释：

- `>>`：追加标准输出
- `2>&1`：把标准错误也重定向到标准输出

例如：

```bash
0 3 * * * /backup.sh >> /var/log/backup.log 2>&1
```

### 10. 权限与安全

要注意：

- 脚本必须可执行

```bash
chmod +x script.sh
```

- 尽量使用绝对路径

```bash
/usr/bin/python3 script.py
```

- 谨慎处理：
    - `sudo`
    - 敏感凭据
    - 高风险命令

### 11. 系统级 cron

除了用户自己的 crontab 之外，还有系统级 cron 配置：

- `/etc/crontab`
- `/etc/cron.daily`
- `/etc/cron.hourly`
- `/etc/cron.weekly`
- `/etc/cron.monthly`

有些脚本也可以直接放进这些目录中。

### 12. 实际应用场景

### 1）机器学习流水线

```bash
0 2 * * * python train.py
```

### 2）数据采集

```bash
*/10 * * * * python scraper.py
```

### 3）系统监控

```bash
* * * * * bash check_cpu.sh
```

### 4）自动清理临时文件

```bash
0 0 * * * rm -rf /tmp/*
```

> 注意：涉及删除命令时一定要非常小心。
> 

### 13. 常见坑

### 1）相对路径

不推荐：

```bash
python script.py
```

推荐：

```bash
/usr/bin/python3 /home/user/script.py
```

### 2）缺少执行权限

### 3）静默失败，没有日志

### 4）时区问题

可以用下面命令查看系统时区：

```bash
timedatectl
```

### 14. 总结

- `cron` 是一个基于时间的任务调度器
- `crontab` 用 5 段时间格式定义任务
- 实战中要特别注意：
    - 使用绝对路径
    - 做好日志重定向
    - 考虑运行环境与终端不同
